## 修正対象と問題点

### 1. `shared/api/index.ts` の `fetchArticles()` が公開 API の `search` パラメータに追随していない
現状:
- `app/api/articles/route.ts` は `search` を受け付けている
- しかし `shared/api/index.ts` の `ArticlesQuery` に `search` がない
- `fetchArticles()` の query string 組み立てにも `search` が含まれていない

問題:
- Web と shared client の機能差が発生する
- iOS/Android 展開時に同一 API 契約を共有できない

対応してほしいこと:
- `shared/api/index.ts` の `ArticlesQuery` に `search?: string` を追加
- `fetchArticles()` のクエリ組み立てに `search` を含める
- 必要なら関連する shared test を追加または更新
- コメントも実装と一致させる

受け入れ条件:
- `fetchArticles()` から `search` が渡せる
- shared client から公開 API の検索仕様を利用できる

---

### 2. `app/api/auth/register/route.ts` に CSRF 保護と abuse 対策がない
現状:
- register API は body validation と email 重複チェックだけ
- `verifyCsrf()` を使っていない
- rate limit もない

問題:
- bot による大量登録
- 外部サイトからの悪用
- 公開 API として防御が弱い

対応してほしいこと:
- `POST /api/auth/register` に CSRF 保護を追加
- register 用 rate limit を追加
- 既存の `lib/ratelimit.ts` を流用できるなら流用
- 必要なら register 専用の rate limiter を追加
- エラーレスポンス形式を既存 API と極力そろえる

受け入れ条件:
- register API が CSRF / abuse 対策を持つ
- 正常系を壊さない
- エラーレスポンスが既存方針と大きくずれない

---

### 3. `lib/csrf.ts` の mobile client 設計と実装が食い違っている
現状:
- コメントでは `allowMobile: true` を使って `X-Client-Platform` + `X-Mobile-Api-Key` を許可する設計
- しかし実装では `Origin` がない場合に無条件通過
- そのため mobile API key を使う設計の意味が薄い
- 実際に `allowMobile: true` を使っているルートも見当たらない

問題:
- セキュリティ境界が曖昧
- コメントと実装がずれている
- 将来 mobile API を増やした時に事故りやすい

ここはどちらかに寄せてください:
A. コメントどおりに厳密化する
- `Origin` がない場合を安易に通さない
- mobile 許可は `allowMobile: true` と API key 条件のときだけ通す

または

B. 現実の運用に合わせてコメント/設計を修正する
- 非ブラウザクライアントを許可する方針なら、その仕様を明文化する
- ただし admin API との境界は明確にする

推奨:
- 中途半端な状態をやめて、コードとコメントを一致させる
- もし breaking になりすぎるなら、安全側に寄せる

対応してほしいこと:
- `lib/csrf.ts` を設計意図に一致させる
- 関連 API の呼び出し側も必要なら調整
- 少なくとも「コメントでは厳しいが実装は素通り」の状態を解消

受け入れ条件:
- mobile 許可経路の仕様がコードで明確
- admin API の保護が弱くならない
- コメントと実装が一致している

---

### 4. `app/(site)/learn/[slug]/page.tsx` の JSON-LD が存在しない `logo.png` を参照している
現状:
- JSON-LD の publisher.logo.url が `${SITE.url}/logo.png`
- しかし `public/logo.png` が存在しない

問題:
- 構造化データ品質が落ちる

対応してほしいこと:

受け入れ条件:
- JSON-LD で存在しない画像 URL を参照しない
- SEO 用メタデータが今より不正確にならない

### 5. 可能なら `tsc` / `vitest` の再現性も改善してほしい
現状:
- `npx tsc --noEmit` が `gray-matter` / `vitest` 解決エラーで落ちる

問題:
- 変更の安全性を担保できない
- CI / 他環境で壊れやすい

対応してほしいこと:
- 原因を特定して、直せるなら直す

レビュー観点:
1. API 契約
   - 通常エラーと SSE ストリーム内エラーの使い分けが自然か

2. 運用性
   - エラー時に常に HTTP 200 を返す設計が監視やログ解析で問題を生まないか

   - 現行 Web クライアントには都合がよくても、将来の iOS/Android / 外部 client で扱いづらくないか

4. エラーハンドリング
   - JSON エラーと SSE エラーの境界が分かりやすいか
   - timeout / provider failure / validation error が適切に分類されているか

- `app/api/ai/route.ts`

2. クライアントは stream 内 `error` で判定するが、監視や API caller には分かりにくい可能性
3. 将来 shared client / mobile client が増えたとき、契約が扱いづらい可能性
