## 1. セキュリティ

### 良い点
- **CSRF対策**: 全APIルートで `Origin` ヘッダーと `x-forwarded-host` の比較検証を実施
- **SQLインジェクション対策**: Drizzle ORMによるパラメータ化クエリを全件使用
- **入力検証**: 全API境界でZodスキーマによるバリデーション
- **レート制限**: Upstash Redis使用、フェイルクローズド（Redis障害時は制限側に倒す）設計
- **XSS対策**: rehype-sanitizeによるMarkdownレンダリング時のサニタイズ

### 改善推奨（中程度）
1. **`app/api/admin/article-bookmarks/route.ts`**: DELETE用のZodスキーマが未定義。`id`パラメータがバリデーションされていない
2. **LIKE検索のエスケープ**: 記事検索で `escapeLike` 関数が未実装。ユーザー入力の `%` `_` がワイルドカードとして解釈される
3. **`app/api/link-preview/route.ts`**: レート制限なし。外部URLをフェッチするエンドポイントのためSSRFリスク
4. **管理画面APIのPUT/DELETE**: `ai-config` のPUT/DELETEにレート制限がかかっていない

### 改善推奨（軽度）
- 管理画面API間で認可チェックのパターンに微妙な不統一がある（一部は `role !== 'admin'` 比較、一部はヘルパー関数使用）

---

## 2. アーキテクチャ

### 良い点
- **3層アーキテクチャ**: repositories → mappers → DTOs の分離が明確
- **Server/Client分離**: Next.js App RouterのServer ComponentとClient Componentの境界が適切
- **関心の分離**: ページコンポーネント、UIコンポーネント、APIルートが整理されたディレクトリ構造

### 改善推奨
1. **APIレスポンス形状の不統一**: 以下の4パターンが混在している
   - `{ data: [...] }` 
   - `{ items: [...], total: number }`
   - `{ ...result }`（スプレッド）
   - 直接オブジェクト返却
   → 統一ラッパー型（例: `ApiResponse<T>`）の導入を推奨
2. **ページネーションAPIの不一致**: オフセットベースとカーソルベースが混在
3. **巨大コンポーネント**:
   - `components/home/ArticleBody.tsx`（899行）— セクションごとに分割推奨
   - `components/chat/AIChatWidget.tsx`（791行）— チャットロジック・UI・状態管理の分離推奨
4. **DTO移行が中途半端**: 一部リポジトリが直接DB rowを返しており、DTOレイヤーが未使用
5. **バレルファイル（index.ts）**: 存在するがインポートに使われていない

---

## 3. パフォーマンス

### 良い点
- **N+1クエリなし**: 関連データをJOINまたは `inArray` で一括取得
- **ISR活用**: 記事ページで `revalidate` と `generateStaticParams` を適切に使用
- **キャッシュ戦略**: Next.jsのフェッチキャッシュとメモ化を活用

### 改善推奨
1. **`next/dynamic` 未使用**: 以下の大型Client Componentが初期バンドルに含まれている
   - `AIChatWidget`（チャット全体）
   - `ArticleBody`（記事本文＋目次＋3Dビューワー）
   - `@google/model-viewer`（3Dモデル表示）
   → `next/dynamic` で遅延ロードし、インタラクションが必要になるまで読み込まない
2. **`next/image` 未使用**: コードベース内に8箇所の `<img>` タグ（`eslint-disable-next-line @next/next/no-img-element` 付き）
   → 自動最適化・遅延ロード・CLS防止のために `<Image>` への置き換えを推奨
3. **エラーバウンダリ**: ルートレベルのみ。チャット・コメント・3Dビューワーなどの独立した機能単位での `error.tsx` がない

---

## 4. 型安全性

### 良い点
- TypeScript strict mode
- Zodによるランタイム型検証と型推論の併用（`z.infer`）
- Drizzleの型安全なクエリビルダー

### 改善推奨
1. **APIレスポンス型のany**: 一部のAPIクライアント呼び出しで `.json()` の戻り値が暗黙的any
2. **`shared.ts` のユーティリティ型**: 一部関数の戻り値型が省略されている
3. **Drizzleのリレーション型**: 一部で `infer` 型より手動のインターフェース定義が使われており、カラム変更時に型不整合のリスク

---

## 5. アクセシビリティ

### 良い点
- ConfirmDialog: `role="dialog"`, `aria-modal`, `aria-labelledby`, `aria-describedby`, フォーカストラップ対応
- セマンティックHTML: 見出し階層・`<nav>`・`<main>` の適切な使用
- `--font-scale` CSS変数によるシニア向け文字サイズ拡大対応

### 改善推奨
1. **`MobileNav` のrole属性誤用**: `role="menu"` が宣言されているが、内部に `menuitem` roleを持つ要素がない。`role="navigation"` が適切
2. **コントラスト比不足**: `#7a6a3d`（ink-3）on `#f3ead8`（paper）= コントラスト比 **3.4:1**（WCAG AAは4.5:1以上が必要）
   - 影響箇所: 補足テキスト、日付表示、プレースホルダー
   - 対策: `#5a4a35` 程度まで暗くする
3. **フォーカス可視化**: 一部のインタラクティブ要素で `:focus-visible` のスタイルが不足（特にカスタムボタン）
4. **スキップリンク**: メインコンテンツへのスキップリンクがない

---

## 6. データベース

### 良い点
- 外部キー制約が適切に設定されている
- 検索・ソート用のインデックスが主要カラムに付与されている
- Neon Serverless + Drizzle ORMの組み合わせがプロジェクト規模に適している

### 改善推奨
1. **updatedAtの自動更新なし**: スキーマに `updatedAt` カラムがあるが、Drizzleの `onUpdateNow()` またはトリガーが設定されておらず、手動更新に依存
2. **マイグレーション履歴の欠落**: `0004` 以降のマイグレーションファイルがリポジトリに存在しない可能性（最新スキーマとマイグレーションの一致確認が必要）
3. **ソフトデリート**: 記事・コメントの削除が物理削除。復旧が必要なケースに備えて `deletedAt` カラムの追加を検討

---

## 7. エラーハンドリング

### 良い点
- APIルートごとに try-catch でエラーをキャッチし、適切なHTTPステータスコードを返している
- 管理画面APIで `error instanceof Error` による型安全なエラーメッセージ抽出

### 改善推奨
1. **グローバルエラーバウンダリ不足**: `app/error.tsx` は存在するが、`app/(site)/error.tsx` などセグメント単位のエラーバウンダリがない
2. **APIエラーレスポンス形式の不統一**: `{ error: string }` / `{ message: string }` / テキスト直接返却 が混在 → `{ error: string; code?: string }` に統一推奨
3. **クライアント側エラー表示**: fetch失敗時のエラー表示が `alert()` に依存している箇所がある（AdminArticleBookmarkTableなど）→ トースト通知またはインラインエラー表示に置き換え推奨
4. **ネットワークエラーのリトライ**: 自動リトライロジックがない

---

## 8. テスト

### 現状
- `vitest` が設定済み
- `scripts/smoke-test.sh` に簡易スモークテストが存在
- **単体テスト・結合テストのファイルがほぼ存在しない**

### 改善推奨
1. **最低限追加すべきテスト**:
   - リポジトリ層のCRUDテスト（インメモリDBまたはテスト用Neonブランチ使用）
   - APIルートのZodバリデーションテスト（正常系＋異常系）
   - 認可チェックの単体テスト（admin/一般ユーザーの分岐）
   - `MarkdownContent` のレンダリングスナップショットテスト
2. **テストヘルパー**: テスト用のファクトリ関数（ユーザー・記事・コメントの作成）を整備
3. **E2Eテスト**: PlaywrightまたはCypressの導入を中期的に検討（認証フロー・記事閲覧・AIチャット）

---

## 9. CSS/デザイン

### 良い点
- familyaidesign casual variantへの統一的な移行が完了
- CSS変数によるデザイントークン一元管理
- 旧Mingeiトークンのエイリアスによる後方互換性の確保

### 改善推奨
1. **ハードコード色の残存**: `#fff`, `#000`, `#E5E7EB`, `#374151` などのハードコードが約15箇所以上に残っている
   - `ConfirmDialog.tsx`: 背景白・ボタン色がハードコード
   - `AdminArticleBookmarkTable.tsx`: テーブルスタイルがハードコード
   - → CSS変数への置き換えを推奨（ダークモード対応の布石にもなる）
2. **アニメーション定義の重複**:
   - `globals.css` と `tailwind.config.ts` の両方に `fadeIn`, `fadeInUp`, `slideUp` などのキーフレームが定義されている
   - → globals.cssに一本化し、tailwind.configでは参照のみにする
3. **未使用CSSクラス**: 旧Mingei時代のクラス（`.card-mingei`, `.btn-mingei` など）が残存している可能性
4. **コンポーネント別CSSの分離**: 一部のコンポーネントでインラインスタイルが大量に使われており、CSS ModulesまたはTailwindユーティリティへの移行が望ましい
