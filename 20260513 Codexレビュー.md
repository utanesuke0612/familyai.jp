# 指摘一覧

## P1: 3D機能のadmin限定公開がAPI側で迂回できる

対象:
`app/api/3d-models/[slug]/route.ts`

問題:
3Dカタログ/詳細ページは `isAdmin()` で隠しているが、`GET /api/3d-models/[slug]` は未認証で公開済みモデルの詳細とアセットURLを返している。

影響:
admin限定プレビューのつもりでも、slugを知っていればAPIからモデル情報を取得できる。公開前の教材・アセット・説明文が漏れる可能性がある。

修正方針:
次のどちらかに統一すること。

1. admin限定プレビューを継続する場合:
   - `GET /api/3d-models/[slug]` にも `isAdmin()` または `requireAdmin()` を入れる。
   - 未認可時は `404` か `403` を返す。UI側が404で隠しているならAPIも404の方が一貫性がある。
   - 3Dアセット配信 `/api/3d-models/assets/...` も、公開済みモデルなら誰でも読める設計でよいのか再確認する。

2. 一般公開にする場合:
   - `app/(site)/tools/ai-kyoshitsu/page.tsx`
   - `app/(site)/tools/ai-kyoshitsu/[slug]/page.tsx`
   - `app/(site)/tools/page.tsx`
   のadmin gate / `adminOnly` を外す。
   - APIコメントも「公開API」として整理する。

優先度:
最優先。公開制御の設計不整合。

## P2: 3D詳細ページのmetadataがadmin認可前に生成される

対象:
`app/(site)/tools/ai-kyoshitsu/[slug]/page.tsx`

問題:
ページ本体はadmin以外 `notFound()` になるが、`generateMetadata` は認可確認前にモデルを取得し、title/description/OGP画像を返している。

影響:
非公開プレビュー扱いの場合、検索botやSNSカード経由でモデル名・説明・サムネイルが露出する。

修正方針:
admin限定を継続するなら、`generateMetadata` 内にも `isAdmin()` を入れる。

例:
```ts
export async function generateMetadata({ params }: PageProps): Promise<Metadata> {
  if (!(await isAdmin())) {
    return {
      title: `見つかりませんでした | ${SITE.name}`,
      robots: 'noindex, nofollow',
    };
  }

  ...
}
```

一般公開にするならこの指摘は不要。ただしP1とセットで方針を統一すること。

## P2: 記事iframeが任意HTTPSをtrusted embed扱いしている

対象:
`components/article/ArticleBody.tsx`

問題:
VOA/YouTube以外の任意HTTPS URLも `trusted-embed` としてiframe表示している。

該当箇所:
`VOA・YouTube 以外の HTTPS URL（S3 等）も信頼済み埋め込みとして扱う`

影響:
管理者投稿前提でも、CMS入力ミスや管理画面侵害時の被害範囲が広がる。意図しない外部サイトを記事内iframeとして表示できる。

修正方針:
allowlist方式に変更する。

推奨:
- VOA公式
- YouTube / youtube-nocookie
- 自社管理のBlob/CDNドメイン
- 必要なら明示的に許可したS3/CloudFrontドメイン

任意HTTPSを許可しない。

加えて、iframeには用途別に以下を検討:
- `sandbox`
- `allow`
- `referrerPolicy`
- `loading="lazy"`

注意:
`srcdoc` はsanitize schema上は許可されているが、現在のparse処理では主に `src` を見ている。不要なら `srcdoc` を許可属性から外す。

## P2: 関連記事取得がORDER BY random()のまま

対象:
`lib/repositories/articles.ts`

問題:
関連記事取得で以下を使用している。

```ts
.orderBy(sql`random()`)
```

影響:
記事数が増えると候補行全体にランダムソートが走り、記事詳細ページのDB負荷が増える。

修正方針:
`random()` を避ける。

推奨案:
1. カテゴリ一致で候補を絞る。
2. `publishedAt DESC`、`viewCount DESC`、`id` などindexが効く順序で取得。
3. 必要なら取得後にアプリ側で少数件だけシャッフル。

例:
```ts
.orderBy(desc(articles.publishedAt))
.limit(limit * 3)
```

その後アプリ側でシャッフルして `limit` 件にする。

## P2: 公開記事検索が `%keyword%` ILIKE でスケールしにくい

対象:
`lib/repositories/articles.ts`
`app/api/articles/route.ts`

問題:
公開記事検索が title/description に対して `%keyword%` ILIKE を使っている。

影響:
通常のbtree indexが効きにくく、記事数が増えると検索APIが重くなる。

修正方針:
Postgres向けに検索indexを追加する。

選択肢:
1. 簡易対応:
   - `pg_trgm` extension
   - `GIN` trigram index
   - `title`, `description` に対する検索高速化

2. 本格対応:
   - `tsvector`
   - 日本語検索要件も含めた全文検索設計

例:
```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;

CREATE INDEX CONCURRENTLY IF NOT EXISTS articles_title_trgm_idx
ON articles USING gin (title gin_trgm_ops);

CREATE INDEX CONCURRENTLY IF NOT EXISTS articles_description_trgm_idx
ON articles USING gin (description gin_trgm_ops);
```

Drizzle migrationで扱う場合、`CONCURRENTLY` は通常migrationと相性が悪い場合があるので、本番適用手順を分けること。

## P2: 同期APIのpagination化に対してClientが先頭200件しか読まない

対象:
`lib/ai-memo-store.ts`
`lib/voaenglish/vocab-store.ts`
`lib/voaenglish/sentence-bookmark-store.ts`

問題:
APIはpagination対応済みだが、Client側は `pageSize=200` を1回だけfetchしている。

例:
```ts
fetch('/api/user/ai-memos?pageSize=200')
fetch('/api/user/vocab-bookmarks?pageSize=200')
fetch('/api/user/sentence-bookmarks?pageSize=200')
```

影響:
201件目以降がマイページやキャッシュに出ない。ユーザーが長期利用するとデータが欠けて見える。

修正方針:
Client側もpaginationを処理する。

最低限:
- `meta.total`
- `meta.page`
- `meta.perPage`

を見て必要ページを追加取得する。

推奨:
将来のiOS/Android同期も考えるなら、offset paginationより以下がよい。

- `cursor`
- `updatedSince`
- `deletedSince`
- 差分同期

段階的対応:
まずWebだけなら、全ページfetch helperを作る。

例:
```ts
async function fetchAllPages<T>(basePath: string): Promise<T[]> {
  const first = await fetch(`${basePath}?page=1&pageSize=200`).then(r => r.json());
  const items = [...first.data];
  const totalPages = Math.ceil(first.meta.total / first.meta.perPage);

  for (let page = 2; page <= totalPages; page++) {
    const next = await fetch(`${basePath}?page=${page}&pageSize=200`).then(r => r.json());
    items.push(...next.data);
  }

  return items;
}
```

## P3: AI EchoはSSE deltaごとにstate更新している

対象:
`components/voaenglish/AIEchoPanel.tsx`

問題:
AIChatWidget側は60ms throttle済みだが、AIEchoPanelは `json.delta` ごとに `patchLevel` している。

該当:
```ts
if (json.delta) {
  accumulated += json.delta;
  patchLevel(lv, { feedback: accumulated });
}
```

影響:

- 60ms throttle
- 終了時に最後の差分をflush
- unmount/abort時にtimer clear
